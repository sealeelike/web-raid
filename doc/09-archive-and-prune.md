# 模块 9：VPS 侧每日归档与保留策略清理

代码：`backup-server-setup/install-archiver.sh`（独立一键脚本，不属于 `backupctl`，在 VPS 上 root 执行）。

## 做什么

给已经在跑的 `rest-server` 追加一层"每日归档 + 按份数淘汰"：

- **每次运行**（systemd timer 每天 03:30 左右触发，也可手动跑）：把 `--private-repos` 下每个客户端的实时仓库目录，用 `rsync --link-dest` 硬链接归档一份到独立目录（默认 `/srv/backup-archive`），文件名是时间戳，不管内容有没有变化都会归档一次
- **归档目录归 root 所有、权限 700**：`rest-server` 的服务账户（`restic-rest-server`）对它没有任何读写权限——这是防"笔记本备份凭据被攻破后连远端历史一起删光"的核心手段，不是靠限制凭据权限（凭据本来就需要完整读写权限才能日常工作），而是把归档层放在这份凭据完全够不到的地方
- **按份数淘汰，不按日期**：固定保留最新 N 份（默认 7），只有真正产生第 N+1 份新归档时才淘汰最老的一份；笔记本长期不在线、不产生新归档，现有的历史不会因为"日期变旧"被误删
- **可选 `chattr +i` 只读保护**：写完一份归档后（见下面"设计上的一个坑"）额外加一层文件系统级不可变属性

这一层完全独立于 `backupctl`：笔记本对归档目录一无所知，也没有任何权限碰它，归档、淘汰全部由 VPS 本地的 systemd timer 自主完成。

## 关键实现点

1. `install-archiver.sh`：交互式部署（归档目录路径、保留份数、是否启用 chattr，回车=默认值/保留原值），生成 `/opt/backup-archiver/archive-and-prune.conf`（纯配置）+ `archive-and-prune.sh`（读配置的通用脚本，不含用户输入直接拼进脚本体，避免把路径字符串直接插值进脚本逻辑）
2. 归档目录首次创建时 `chown root:root && chmod 700`；重新运行脚本可以调整保留份数等参数，不会丢已有归档（沿用同一份 conf 文件，读出旧值当新的默认值）
3. `archive-and-prune.sh` 遍历 `$DATA_DIR/*/`（`--private-repos` 懒创建的每用户仓库目录，`.htpasswd` 这类平级文件因为没有末尾 `/` 匹配不上 glob，天然被跳过），对每个用户目录独立走一遍"归档 → 可选加锁 → 按份数淘汰"
4. systemd：`backup-archiver.service`（oneshot，`ProtectSystem=strict` + `ReadOnlyPaths=$DATA_DIR` + `ReadWritePaths=$ARCHIVE_ROOT` 加固）+ `backup-archiver.timer`（`OnCalendar=03:30`，`Persistent=true`，`RandomizedDelaySec=600` 避免多台 VPS/多个用户同一时刻扎堆）
5. **归档层用 root 而不是另开一个专用低权限系统用户执行**：这是刻意的简化。归档目录本身"不归 `restic-rest-server` 所有"这个隔离目标，靠的是目录所有权是 root（而不是一个新建的专用用户），跟归档脚本自己用什么身份跑没有直接关系——反而如果归档脚本用普通用户跑，还得额外解决"这个新用户怎么有权限读到 `restic-rest-server` 账户 0700 权限下的仓库文件"这个问题（要么把它加进 `restic-rest-server` 组，等于又开了一个新的信任关系）。这套系统真正要防的是"笔记本凭据泄露"，不是"VPS root 被攻破"（后者不管怎么设计归档层身份都防不住，attacker 已经是 root 了），所以归档脚本用 root 跑是合理的权衡，见 `doc/08-multi-device-same-vps.md` 里"隔离的性质"一节的同一个论证

## 设计上的一个坑：chattr +i 会悄悄弄坏硬链接去重

实测过程中发现一个真实 bug（不是理论推测）：最初的实现是"归档一创建就立刻 `chattr +i` 这一份"，结果第二天再跑，`rsync --link-dest` 引用昨天那份归档时完全没有产生硬链接（`ls -i` 看到的 inode 号不一样，`stat` 的链接数也是 1），归档实际上悄悄退化成了每天全量复制——`chattr(1)` 文档写得很明确："immutable 的文件不能被链接"（no link can be created to this file），`rsync --link-dest` 内部就是对目标文件调用 `link()` 系统调用，对一个已经 `+i` 的源文件调用 `link()` 会直接失败，rsync 不会报错中止，只是静默回退成整份复制，不看日志根本发现不了。

**修复**：`chattr +i` 只加到"上一份"归档（这一轮 rsync 已经用完它当 link-dest 源之后），永远不加到刚创建的这一份——保证任何时刻"最新一份"归档都是可写/可链接状态，供下一轮引用；较旧的归档（第二新及更早）全部处于加锁状态。这样两个目标都不牺牲：绝大多数归档份数都有不可变保护，硬链接去重也一直正常工作。

这个修复带来一个可以接受的次要影响：淘汰最老一份归档时，需要先 `chattr -R -i` 解锁才能 `rm -rf`；如果被删的文件恰好和更新一份归档共享同一个硬链接 inode（内容没变过），解锁这一步会连带让那个共享 inode 暂时失去不可变属性（chattr 的不可变位是 inode 级别的，不是路径级别的），直到下次归档循环里它被下一轮的"锁定上一份"逻辑重新加锁为止。这个窗口期只有在"VPS root 被攻破"这个本来就不在防护范围内的场景下才有意义（`restic-rest-server` 账户全程没有任何权限碰归档目录，不管 chattr 状态如何），所以接受这个次要影响，不为此引入更复杂的机制（比如给每个 inode 单独追踪引用计数）。

## 实测记录（2026-07-04，真实测试 VPS，临时实例）

1. 用真实数据搭建测试环境：VPS 上临时装一个 `rest-server` 实例（端口 9194，用户 `archtest`），本机 `backupctl target add` 对接、`path add` 一个含文本+二进制随机内容的测试目录、`backupctl run --force` 产生一份真实的加密快照
2. **第一轮 bug 复现**：装第一版 `install-archiver.sh`（归档后立即 `chattr +i` 自己）跑两轮归档，`stat`/`ls -i` 确认两份归档里同名的 `config` 文件 inode 不同、链接数均为 1——证实硬链接没有生效，归档已经退化成全量复制
3. 定位到 `chattr(1)` 关于 immutable 文件"不能被链接"的机制，修复脚本逻辑为"锁上一份、留最新一份可写"，重新部署（`chattr -R -i` 解锁旧测试数据 → 删除重来）
4. **第二轮验证（修复后）**：连续跑 4 轮归档（相邻两轮间隔 1~1.5 秒），每轮之间用 `find`/`stat` 确认：
   - 相邻两份归档里内容没变的文件（比如 `config`）inode 相同、链接数为 2，证明硬链接确实生效
   - `lsattr -d` 确认永远只有"上一份"被打上 `----i-----------------`，"最新一份"始终是空标志（可写可链接）
5. **按份数淘汰验证**（`KEEP_COUNT=2`）：第 3、4 轮归档产生后，目录里确实只剩最新 2 份，最老的被自动删除，日志打印了具体淘汰的路径
6. **权限隔离验证**：`su -s /bin/sh restic-rest-server -c 'ls /srv/backup-archive-test/archtest/'` 和 `cat` 归档里的 `config` 文件均返回 `Permission denied`——确认 `rest-server` 的服务账户对归档目录确实没有任何访问权限
7. **可恢复性验证**（不是理论推测，实际做了一次真实恢复）：把最新一份归档整体 `rsync` 拉回笔记本本地，用本地 `restic` 直接指向这份归档路径（`-r /tmp/archive-verify`），`restic snapshots` 成功列出快照，`restic restore` 成功恢复文件，`diff`/`cmp` 确认恢复出的文件内容和原始测试文件逐字节一致
8. **"长期不产生新归档不会误删历史"这一条**：脚本的淘汰逻辑通篇没有任何基于日期/`mtime` 的判断，唯一的淘汰触发条件是"这一轮又新产生了一份归档、且总数超过保留份数"——没有代码路径会在"单纯时间流逝、没有新归档产生"时删除任何东西，这是可以直接从代码逻辑上确认的结构性保证，不需要额外伪造系统时间去模拟"过了很久"
9. 测试完毕后完全清理：VPS 上停用/删除 `backup-archiver` 的 service/timer、解锁并删除测试归档目录和 `/opt/backup-archiver`、停用删除临时 `restic-rest-server` 实例（服务、用户、安装目录、`/root/` 下的脚本副本），确认端口 9194 已释放；本机删除测试 target/path 配置和临时文件

## 已知待办 / 注意事项

- 保留份数目前是全局参数（`archive-and-prune.conf` 里的 `KEEP_COUNT`），对该 VPS 上所有客户端一视同仁；如果将来想给不同客户端设置不同的保留份数，需要扩展成按用户名的配置项，当前场景不需要
- 归档层跑在 root 下（见上面"关键实现点"第 5 条的论证），如果以后有更高的隔离诉求（比如防止归档脚本自身被利用去提权），可以改成专用用户 + 让它加入一个只读的 ACL/组权限，但当前场景下增加的复杂度和收益不成比例
- 淘汰"按份数"目前是固定滑动窗口（保留最新 N 份），原始设计里提到的祖父-父亲-儿子式阶梯保留（比如"最近 7 天每天一份 + 每周一份 + 每月一份"）是特意留到以后再做的更复杂模式，当前按用户确认先做固定滚动窗口
